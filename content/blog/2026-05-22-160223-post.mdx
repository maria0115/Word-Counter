안녕하세요, 존경하는 동료 개발자 여러분! 여러분의 백엔드 아키텍처를 견고하고 빠르게 만드는 데 열정적인 블로거입니다. 오늘은 우리가 매일 사용하는 JPA(Java Persistence API) 환경에서 종종 간과되지만, 애플리케이션의 성능을 송두리째 잡아먹는 **N+1 문제**에 대해 깊이 파고들어 보겠습니다.

JPA는 객체 지향적인 방식으로 데이터베이스와 상호작용할 수 있게 해주어 생산성을 비약적으로 높여줍니다. 하지만 그 편리함 뒤에는 복잡한 데이터 로딩 전략이 숨어 있으며, 이를 제대로 이해하지 못하면 애플리케이션은 예상치 못한 성능 병목에 시달릴 수 있습니다. 그중 가장 대표적인 것이 바로 N+1 문제입니다.

자, 그럼 이 성능 저하의 주범을 어떻게 찾아내고, 어떻게 해결할 수 있을지 함께 탐구해 볼까요?

## 📚 N+1 문제, 과연 무엇인가?

N+1 문제는 엔티티 관계(특히 `OneToMany`, `ManyToOne` 등)에서 **지연 로딩(Lazy Loading)** 전략을 사용할 때 발생하기 쉽습니다. 간단히 말해, **하나의 엔티티를 조회하기 위한 쿼리(1)** 후에, 해당 엔티티와 연관된 **컬렉션 또는 다른 엔티티를 조회하기 위해 추가적인 N개의 쿼리가 발생하는 현상**을 말합니다.

예를 들어, 사용자 목록을 조회하면서 각 사용자가 주문한 주문 내역을 함께 보여줘야 하는 시나리오를 생각해 봅시다. JPA는 기본적으로 `@OneToMany` 관계를 지연 로딩으로 설정합니다. 따라서 사용자 목록을 조회할 때 (1개의 쿼리), 각 사용자의 주문 목록을 실제로 접근하는 순간(N번의 루프), 사용자 수(N)만큼 주문 조회 쿼리가 추가로 발생하게 됩니다.

이러한 N+1 쿼리는 데이터베이스 네트워크 부하를 증가시키고, 불필요한 데이터베이스 접근을 야기하여 애플리케이션의 응답 속도를 현저히 떨어뜨립니다.

### 📝 N+1 문제 시나리오 예시

다음과 같은 `User`와 `Order` 엔티티가 있다고 가정해 봅시다.

```java
// User.java
import jakarta.persistence.*;
import java.util.ArrayList;
import java.util.List;

@Entity
@Table(name = "users") // SQL 키워드 user와의 충돌 방지
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;

    // OneToMany 관계는 기본적으로 LAZY 로딩입니다.
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders = new ArrayList<>();

    // 기본 생성자
    protected User() {}

    public User(String name) {
        this.name = name;
    }

    // Getters and Setters
    public Long getId() { return id; }
    public String getName() { return name; }
    public List<Order> getOrders() { return orders; }

    public void addOrder(Order order) {
        this.orders.add(order);
        order.setUser(this);
    }
}
```

```java
// Order.java
import jakarta.persistence.*;

@Entity
@Table(name = "customer_orders") // SQL 키워드 order와의 충돌 방지
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String orderNumber;
    private int amount;

    // ManyToOne 관계는 기본적으로 EAGER 로딩입니다. 하지만 N+1 문제 예시를 위해 LAZY로 설정합니다.
    // 실무에서는 ManyToOne도 LAZY로 설정하고 필요할 때 Fetch Join을 쓰는 것이 좋습니다.
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;

    // 기본 생성자
    protected Order() {}

    public Order(String orderNumber, int amount) {
        this.orderNumber = orderNumber;
        this.amount = amount;
    }

    // Getters and Setters
    public Long getId() { return id; }
    public String getOrderNumber() { return orderNumber; }
    public int getAmount() { return amount; }
    public User getUser() { return user; }
    public void setUser(User user) { this.user = user; }
}
```

이제 이 엔티티들을 사용하여 N+1 문제가 발생하는 시나리오를 재현해 봅시다.

```java
// UserRepository.java
import org.springframework.data.jpa.repository.JpaRepository;
import java.util.List;

public interface UserRepository extends JpaRepository<User, Long> {
    // 추가적인 쿼리 메소드가 필요할 때 정의합니다.
}
```

```java
// UserService.java
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

@Service
@Transactional(readOnly = true)
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // N+1 문제가 발생하는 메소드
    public List<User> getAllUsersWithOrdersNPlus1() {
        System.out.println("--- N+1 문제 발생 시나리오 시작 ---");
        List<User> users = userRepository.findAll(); // 1. 모든 User를 조회하는 쿼리 발생

        // 각 User의 order 컬렉션에 접근할 때마다 추가 쿼리 발생
        // (프록시 초기화 시점)
        for (User user : users) {
            System.out.println("User: " + user.getName() + ", Orders Count: " + user.getOrders().size()); // N. 각 User의 Orders를 조회하는 쿼리 발생
        }
        System.out.println("--- N+1 문제 시나리오 종료 ---");
        return users;
    }
}
```

`application.properties`에 SQL 로깅을 활성화하여 어떤 쿼리가 발생하는지 확인해 봅시다.

```properties
spring.jpa.properties.hibernate.format_sql=true
spring.jpa.properties.hibernate.use_sql_comments=true
spring.jpa.show-sql=true
logging.level.org.hibernate.SQL=DEBUG
logging.level.org.hibernate.orm.jdbc.bind=TRACE
```

이 코드를 실행하면 다음과 유사한 쿼리 패턴을 볼 수 있습니다 (콘솔 출력).

1.  `SELECT u1_0.id, u1_0.name FROM users u1_0` (모든 `User` 조회)
2.  `SELECT o1_0.user_id, o1_0.id, o1_0.amount, o1_0.order_number FROM customer_orders o1_0 WHERE o1_0.user_id=?` (첫 번째 `User`의 `Order` 조회)
3.  `SELECT o1_0.user_id, o1_0.id, o1_0.amount, o1_0.order_number FROM customer_orders o1_0 WHERE o1_0.user_id=?` (두 번째 `User`의 `Order` 조회)
4.  ... N번 반복

사용자가 100명이라면 총 101개의 쿼리가 실행되는 비효율적인 상황이 발생합니다!

## 🛠️ N+1 문제 해결 전략

이제 N+1 문제를 효과적으로 해결할 수 있는 다양한 전략들을 살펴보겠습니다. 각 전략은 장단점이 있으므로, 상황에 맞게 적절히 선택하는 것이 중요합니다.

### 1. `FETCH JOIN` 활용하기

`FETCH JOIN`은 JPQL(Java Persistence Query Language)에서 연관된 엔티티를 한 번의 쿼리로 함께 로딩하도록 지시하는 강력한 방법입니다.

```java
// UserRepository.java
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import java.util.List;

public interface UserRepository extends JpaRepository<User, Long> {

    // JPQL의 FETCH JOIN을 사용하여 User와 연관된 Orders를 한 번에 가져옵니다.
    // DISTINCT 키워드는 OneToMany 관계에서 발생할 수 있는 중복 데이터를 제거합니다.
    @Query("SELECT DISTINCT u FROM User u JOIN FETCH u.orders")
    List<User> findAllWithOrdersFetchJoin();
}
```

```java
// UserService.java (추가 메소드)
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

@Service
@Transactional(readOnly = true)
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<User> getAllUsersWithOrdersFetchJoin() {
        System.out.println("--- FETCH JOIN 시나리오 시작 ---");
        List<User> users = userRepository.findAllWithOrdersFetchJoin(); // 1개의 쿼리로 User와 Orders 함께 조회

        for (User user : users) {
            System.out.println("User: " + user.getName() + ", Orders Count: " + user.getOrders().size()); // 이미 로딩되었으므로 추가 쿼리 없음
        }
        System.out.println("--- FETCH JOIN 시나리오 종료 ---");
        return users;
    }
}
```

**장점**:
*   단 한 번의 SQL 쿼리로 연관된 데이터를 모두 로딩하여 N+1 문제를 완벽히 해결합니다.
*   가장 일반적이고 강력한 해결책입니다.

**단점**:
*   `OneToMany` 또는 `ManyToMany` 관계에서 `FETCH JOIN`을 사용하면 결과 집합에 중복된 엔티티가 포함될 수 있습니다. 이를 막기 위해 `DISTINCT` 키워드를 사용해야 합니다.
*   둘 이상의 컬렉션(`@OneToMany`)에 `FETCH JOIN`을 적용하면 **카테시안 곱(Cartesian Product)**이 발생하여 쿼리 결과가 예상보다 훨씬 커지고 메모리 문제가 발생할 수 있습니다. (예: `User` - `Orders`, `User` - `Reviews`를 동시에 `FETCH JOIN`) 이 경우, 여러 쿼리로 나누거나 `@EntityGraph` 또는 DTO 프로젝션을 고려해야 합니다.

### 2. `@EntityGraph` 활용하기

`@EntityGraph`는 `FETCH JOIN`과 유사하게 연관 엔티티를 함께 로딩하지만, JPQL 쿼리 없이 어노테이션을 통해 선언적으로 fetch 전략을 정의할 수 있게 해줍니다. Spring Data JPA와 함께 사용하기에 매우 편리합니다.

```java
// UserRepository.java (추가 메소드)
import org.springframework.data.jpa.repository.EntityGraph;
import org.springframework.data.jpa.repository.JpaRepository;
import java.util.List;

public interface UserRepository extends JpaRepository<User, Long> {

    // @EntityGraph를 사용하여 User와 Orders를 한 번에 가져옵니다.
    @EntityGraph(attributePaths = {"orders"})
    List<User> findAllWithOrdersEntityGraph();

    // 여러 개의 NamedEntityGraph를 엔티티에 정의하여 사용할 수도 있습니다.
    // @NamedEntityGraph(name = "User.withOrders", attributeNodes = @NamedAttributeNode("orders"))
    // User 엔티티에 위 어노테이션을 추가하고, 아래와 같이 사용할 수 있습니다.
    // @EntityGraph(value = "User.withOrders")
    // List<User> findAllWithOrdersNamedEntityGraph();
}
```

```java
// UserService.java (추가 메소드)
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

@Service
@Transactional(readOnly = true)
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<User> getAllUsersWithOrdersEntityGraph() {
        System.out.println("--- @EntityGraph 시나리오 시작 ---");
        List<User> users = userRepository.findAllWithOrdersEntityGraph(); // 1개의 쿼리로 User와 Orders 함께 조회

        for (User user : users) {
            System.out.println("User: " + user.getName() + ", Orders Count: " + user.getOrders().size()); // 이미 로딩되었으므로 추가 쿼리 없음
        }
        System.out.println("--- @EntityGraph 시나리오 종료 ---");
        return users;
    }
}
```

**장점**:
*   JPQL을 직접 작성하지 않고 선언적으로 fetch 전략을 정의할 수 있어 코드가 깔끔하고 가독성이 좋습니다.
*   동적으로 fetch 전략을 변경해야 할 때 유용합니다. (예: 뷰에 따라 다르게 로딩)

**단점**:
*   `FETCH JOIN`과 마찬가지로 여러 컬렉션을 동시에 `@EntityGraph`로 지정하면 **카테시안 곱**이 발생할 수 있습니다.
*   `FETCH JOIN`만큼 유연하지 않은 경우도 있습니다 (예: 특정 조건으로 필터링된 연관 엔티티만 가져오는 등).

### 3. DTO 프로젝션 활용하기

특정 데이터를 읽어오는 목적에만 집중하여, JPA 엔티티가 아닌 DTO(Data Transfer Object)로 직접 결과를 매핑하는 방법입니다. 필요한 컬럼만 선택적으로 조회하므로 네트워크 및 메모리 사용량을 최적화할 수 있습니다.

```java
// UserOrderDTO.java
// 필요한 데이터만 담는 DTO (Java 16+ record 사용)
public record UserOrderDTO(
    Long userId,
    String userName,
    Long orderId,
    String orderNumber,
    int amount
) {
    // DTO 생성자 (JPQL new 키워드에 사용될 것)
    public UserOrderDTO(Long userId, String userName, Long orderId, String orderNumber, int amount) {
        this.userId = userId;
        this.userName = userName;
        this.orderId = orderId;
        this.orderNumber = orderNumber;
        this.amount = amount;
    }
}
```

```java
// UserRepository.java (추가 메소드)
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import java.util.List;

public interface UserRepository extends JpaRepository<User, Long> {

    // JPQL의 'new' 키워드를 사용하여 UserOrderDTO로 직접 프로젝션합니다.
    @Query("SELECT new com.example.blog.dto.UserOrderDTO(u.id, u.name, o.id, o.orderNumber, o.amount) " +
           "FROM User u JOIN u.orders o")
    List<UserOrderDTO> findAllUserOrderDTOs();
}
```

```java
// UserService.java (추가 메소드)
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

@Service
@Transactional(readOnly = true)
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<UserOrderDTO> getAllUserOrderDTOs() {
        System.out.println("--- DTO 프로젝션 시나리오 시작 ---");
        List<UserOrderDTO> dtos = userRepository.findAllUserOrderDTOs(); // 1개의 쿼리로 DTO 형태로 조회

        for (UserOrderDTO dto : dtos) {
            System.out.println("User ID: " + dto.userId() + ", Order Number: " + dto.orderNumber());
        }
        System.out.println("--- DTO 프로젝션 시나리오 종료 ---");
        return dtos;
    }
}
```

**장점**:
*   필요한 데이터만 정확히 조회하므로 가장 높은 성능 효율을 제공할 수 있습니다.
*   읽기 전용 뷰(Read-Only View)나 리포트성 데이터 조회에 이상적입니다.
*   JPA 엔티티의 관리 오버헤드가 발생하지 않습니다.

**단점**:
*   엔티티를 직접 사용하는 것과 달리, 조회된 DTO는 영속성 컨텍스트의 관리 대상이 아닙니다. 따라서 DTO를 수정한 후 `save()` 등의 작업을 하려면 별도의 변환 과정이 필요합니다.
*   컬렉션을 포함하는 복잡한 DTO 구조를 단일 쿼리로 매핑하기는 쉽지 않습니다. (이 경우, 여러 쿼리를 사용하거나 `@EntityGraph`와 함께 DTO 변환 로직을 사용하는 것이 좋습니다.)

### 4. Batch Fetching (Hibernate 특화)

Hibernate는 `default_batch_fetch_size` 설정 또는 `@BatchSize` 어노테이션을 통해 N+1 쿼리를 N/K+1 쿼리로 줄이는 기능을 제공합니다. 이는 N개의 개별 쿼리를 K개 단위로 묶어 실행하는 방식입니다. 완벽한 1개의 쿼리는 아니지만, N+1보다는 훨씬 효율적입니다.

```java
// User.java (BatchSize 어노테이션 추가)
import jakarta.persistence.*;
import java.util.ArrayList;
import java.util.List;
import org.hibernate.annotations.BatchSize; // Hibernate 어노테이션

@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;

    // BatchSize를 사용하여 orders 컬렉션을 10개 단위로 한 번에 가져옵니다.
    @BatchSize(size = 10) // LAZY 로딩 시 적용됩니다.
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders = new ArrayList<>();

    // ... (생성자, Getter/Setter 동일)
}
```

또는 `application.properties`에 전역 설정을 할 수도 있습니다.

```properties
# application.properties
spring.jpa.properties.hibernate.default_batch_fetch_size=100
```

`BatchSize`는 `LAZY` 로딩과 함께 사용될 때 빛을 발합니다. 첫 `User` 목록 로딩 시 N+1 쿼리가 발생해야 하는 상황에서, `BatchSize` 설정 덕분에 Hibernate는 특정 수의 엔티티 ID를 모아 `IN` 절을 사용하여 여러 엔티티의 연관 관계를 한 번에 조회합니다.

```java
// UserService.java (Batch Fetching 테스트 메소드)
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

@Service
@Transactional(readOnly = true)
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // BatchSize 설정 후, N+1 문제가 개선되는 것을 확인하는 메소드
    public List<User> getAllUsersWithOrdersBatchFetching() {
        System.out.println("--- Batch Fetching 시나리오 시작 ---");
        List<User> users = userRepository.findAll(); // 1. 모든 User 조회

        // 각 User의 order 컬렉션에 접근할 때, BatchSize에 따라 묶어서 쿼리 발생
        for (User user : users) {
            System.out.println("User: " + user.getName() + ", Orders Count: " + user.getOrders().size());
        }
        System.out.println("--- Batch Fetching 시나리오 종료 ---");
        return users;
    }
}
```

**장점**:
*   코드 변경을 최소화하면서 N+1 문제를 완화할 수 있습니다.
*   전역 설정으로 적용할 수 있어 편리합니다.

**단점**:
*   여전히 N개의 쿼리가 아닌 N/K개의 쿼리가 발생하므로, 완벽한 단일 쿼리 솔루션은 아닙니다.
*   배치 사이즈를 너무 크게 설정하면 메모리 문제가 발생할 수 있고, 너무 작게 설정하면 효과가 미미할 수 있습니다. 적절한 튜닝이 필요합니다.

## 🕵️‍♂️ N+1 문제 감지 및 모니터링

N+1 문제를 사전에 예방하는 것이 최선이지만, 발생했을 때 빠르게 감지하는 것도 중요합니다.
*   **SQL 로깅**: `spring.jpa.show-sql=true`, `logging.level.org.hibernate.SQL=DEBUG` 설정을 통해 애플리케이션이 실행하는 모든 SQL 쿼리를 확인하여 N+1 패턴을 식별할 수 있습니다.
*   **p6spy 또는 datasource-proxy**: 실제 프로덕션 환경에서는 성능 오버헤드 없이 SQL 쿼리 및 실행 시간을 로깅해주는 도구를 사용하는 것이 좋습니다. 이들은 JDBC 드라이버를 래핑하여 모든 쿼리를 가로채고 통계를 제공합니다.

## 💡 결론 및 최적화 전략

JPA N+1 문제는 백엔드 애플리케이션의 성능을 저하시키는 주요 원인 중 하나입니다. 하지만 위에서 설명한 다양한 해결책들을 통해 효과적으로 대응할 수 있습니다.

**핵심 권장 사항**:
1.  **기본적으로 `@OneToMany` 및 `@ManyToMany`는 `LAZY` 로딩으로 설정하세요.** `EAGER` 로딩은 예상치 못한 대량의 데이터를 로딩하게 하여 더 큰 문제를 야기할 수 있습니다.
2.  **`FETCH JOIN` 또는 `@EntityGraph`를 적극 활용하세요.** 특정 사용 사례에서 연관 엔티티가 반드시 필요한 경우, N+1 문제를 해결하는 가장 직접적인 방법입니다.
3.  **읽기 전용 데이터에는 DTO 프로젝션을 고려하세요.** 엔티티 관리의 오버헤드 없이 필요한 데이터만 가져올 때 최적의 성능을 제공합니다.
4.  **`Batch Fetching`은 차선책이자 보완책입니다.** 모든 `FETCH JOIN`이 불가능하거나 복잡할 때, 혹은 미처 발견하지 못한 N+1 문제를 완화하는 데 도움을 줍니다.
5.  **항상 SQL 쿼리를 모니터링하세요.** 개발 단계에서부터 SQL 로깅을 통해 쿼리 패턴을 확인하고, 필요하다면 프로덕션 환경에서도 성능 모니터링 도구를 활용하여 병목 현상을 파악해야 합니다.

성능 최적화는 단순히 코드를 빠르게 만드는 것을 넘어, 사용자 경험을 향상시키고 시스템 자원을 효율적으로 사용하는 중요한 과정입니다. JPA의 강력함을 제대로 활용하기 위해서는 이러한 깊은 이해가 필수적입니다.
